# Alipay 插件差异说明（基于 alipay-go 现状）

更新时间：2026-03-21  
适用范围：
- `payment/plugins/alipay-go`

## 1. 当前已实现能力（代码已落地）

- 官方版单插件：当前仅维护 `alipay-go`（支付宝官方支付），不再混入 `alipayd/alipaysl` 模式分支。
- 支付方式覆盖：`biztype_alipay` 的 `1~8` 都已接入对应分支调度（页面支付、手机支付、扫码、当面付、预授权、应用支付、JSAPI、订单码）。
- OAuth 买家身份闭环：
  - 在当面付/JSAPI 分支会读取 `auth_code`；
  - 无 `auth_code` 且在支付宝环境时会发起授权跳转；
  - 有 `auth_code` 时会换取买家身份并注入下单参数。
- 订单通知：普通支付通知（`TRADE_SUCCESS/TRADE_FINISHED`）验签后可完成订单。
- 业务能力：已具备订单查询、退款、退款查询、转账、转账查询、余额查询。

## 2. 当前真实缺口（按代码核对）

### 2.1 预授权订单状态链路未完整打通

- 预授权下单已实现（冻结下单）。
- 但通知侧只有通用 `notify`，按普通交易字段处理（如 `out_trade_no/trade_status`），没有预授权专用通知动作。
- 查询侧也只实现了普通交易查询、退款查询、转账查询，未看到预授权专用状态查询与状态映射。

结论：`mode=5` 具备“发起能力”，但缺“通知 + 查询 + 状态推进”的完整闭环。

### 2.2 缺少直付通合单支付能力

- `alipay-go` 当前定位是官方普通版能力，不包含直付通合单（combine）链路。
- 对比 epay 的 `alipayd`，当前仍缺：合单下单、合单回调、合单查询、合单状态处理。

## 3. 后续改造建议

1. 先补齐预授权状态链路：
- 增加预授权专用通知处理；
- 增加预授权状态查询；
- 打通预授权下单、通知、查询到订单状态的统一映射。

2. 再补直付通合单能力：
- 建议独立到 `alipayd-go` 插件实现，避免污染 `alipay-go` 官方版职责边界；
- 覆盖 combine 下单、回调、查询、状态映射全链路。

## 4. 其他边界说明

- 证书模式当前不在本插件范围（仅公钥模式）。
- SDK 中转页机制保持不变：插件返回非 `jump` 页面时由内核中转。

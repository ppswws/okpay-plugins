###1.接口规则
测试请求地址	https://test.trx.helipay.com/trx/app/interface.action
生产请求地址	https://pay.trx.helipay.com/trx/app/interface.action
传输方式	采用HTTP传输
提交方式	采用POST方法提交
字符编码	统一采用UTF-8字符编码
签名算法	MD5
签名要求	请求和接收数据在要求必签名的均需要校验签名,排除签名参数不签名

###2.签名规则
①国密 签名
无论是请求还是应答，签名原始串按以下方式组装成字符串：
1、除 sign 参数值外，其他需参与签名的参数值按照参数名的顺序（&P1参数值&P2参数值&P3参数值…）进行拼接。
2、签名原始串中，字段值都采用原始值，中文不需要URL Encode编码。
3、参数值为空也需要拼接签名(&P1参数值&P2参数值&P3参数值&&P5参数值…)，除接口中明确规定不参与签名的参数值。
4、同步响应签名时，传送的sign参数不参与签名，其他参数按照规则拼接待签名串，使用平台公钥进行SM3WITHSM2验签。

参数原文串拼接示例：

假设提交的参数值如下:
P1_bizType : AppPay
P2_orderId : 20170302191815
P3_customerNumber : C1800000002
P4_payType : SCAN
P5_orderAmount : 0.01
P6_currency : CNY
P7_authcode : 0
P8_appType : WXPAY
P9_notifyUrl : http://localhost
P10_successToUrl : http://localhost
P11_orderIp : 127.0.0.1
P12_goodsName : 商品名称
P13_goodsDetail :
P14_desc :备注

拼接的原文串如下：
&AppPay&p_20170302191815&C1800000002&SCAN&0.01&CNY&0&WXPAY&http://localhost&http://localhost&127.0.0.1&商品名称&&备注

②MD5 签名
MD5 是一种摘要生成算法，通过在签名原始串后加上商户通信密钥的内容，进行 MD5 运算，形成的摘要字符串即为签名结果。
注意：签名时将字符串转化成字节流时指定的编码字符集应与参数 charset 一致。

MD5 签名计算公式：
sign = Md5(原文串&商户密钥)

签名示例：
假设商户密钥如下：
rV8u3c2n2hlTCIDWyzei7iz66DiQlYTH
拼接的原文串加上商户密钥（原文串&rV8u3c2n2hlTCIDWyzei7iz66DiQlYTH）进行MD5计算，结果如下：
sign = 84533392d768af905bebc19ddf932f47

###3.加密、数字签名
加密：对敏感信息字段进行 SM4 算法加密，对称加密的秘钥 Key 由商户生成随机 128 比特流构成，然后使用平台公钥（由平台提供）对 Key 进行 SM2WITHSM3 加密，交易时上送
签名：使用商户私钥进行SM3WITHSM2签名，平台使用商户公钥进行验签；响应时服务端使用平台私钥进行SM3WITHSM2签名，商户使用平台公钥进行验签。

###3.1 签名算法
SM3WITHSM2：国密算法签名

证书**密钥**获取：
登录商户后台，进行证书申请操作，在页面下载证书，从证书获取私钥，具体操作请参考《商户后台证书下载指引.doc》
签名使用说明：
商户拿到请求时的待签名字符串后，把待签名字符串与商户的私钥一同放入 SM2 签名方法中进行签名运算，从而得到签名结果字符串。

待签名字符串：
&AppPay&p_20170302191815&C1800000002&SCAN&0.01&CNY&0&WXPAY&http://localhost&http://localhost&127.0.0.1&商品名称&&备注

签名方法：
签名串 = SM2（待签名字符串，商户SM2私钥）。
计算**签名串(sign)：**
MEUCIQDRhEdGQfim/8e9yMwRa9L4ybbgf1IWLH8QIC1FuepAtAIgdar2LYZmbz+1ubZsYjakoUm/4RPRfdDVIBJ+IIPR1cU=
备注：签名串的生成详情可参考各版本demo中签名方法。

###3.2 加解密算法
SM3WITHSM2：国密SM4算法
接口参数中包含银行卡等敏感信息，为保证敏感信息的安全传输，需要对敏感信息使用SM4/CBC/PKCS7Padding 进行对称加密。
1，请求时商户随机生成的加密密钥sm4 key需要使用平台公钥进行SM2加密后通过encryptionKey字段上送给平台。
2，平台返回时随机生成加密密钥sm4 key，使用商户公钥进行SM2加密后通过encryptionKey字段响应，商户使用自己的私钥解密。
CBC模式要求加解密时使用同样的向量（IV）值，故双方固定 IV 值为：AQ4Zvt54xKn9QaW86ZzWdg== （base64 格式，使用前需解码）


###4. 扫码、刷卡下单接口
应用场景

主扫:商户通过本接口将订单信息上送至合利宝，合利宝调用微信或支付宝下单接口生成二维码返回给商户，商户将二维码信息展示给用户，由用户扫描二维码完成订单支付。

刷卡(被扫): 用户展示微信钱包，支付宝内的“刷卡条码/二维码”给商户系统(商户扫描枪等工具)扫描后直接完成支付的模式。商户调用合利宝刷卡接口,主要应用线下面对面收银的场景。

###5.支付交易结果异步通知
用户支付完成后，合利宝以post请求方式把扫码支付结果推送给商户系统，商户需要接收处理，并返回应答（响应success字符串）。
重发机制
对后台通知交互时，如果收到商户的应答不是成功或超时，合利宝认为通知失败，合利宝重新发起通知，五次回调通知间隔时间为15秒、15秒、30秒、180秒、3600秒；尽可能提高通知的成功率，但不保证通知最终能成功。
由于存在重新发送后台通知的情况，因此同样的通知可能会多次发送给商户系统。商户系统必须能够正确处理重复的通知。
推荐的做法是，当收到通知进行处理时，首先检查对应订单的状态，判断该通知是否已经处理过，如果没有处理过再进行处理，如果处理过直接返回结果成功。
特别注意：商户后台接收到支付结果通知参数后，要对接收到通知参数里的订单号rt5_orderId和订单金额rt10_orderAmount和自身业务系统的订单和金额做校验，校验一致后才更新数据库订单状态。

通知地址
该地址是通过【下单接口】中请求参数P{x}_notifyUrl设置，通知url必须可直接访问的url,不可以携带参数。
